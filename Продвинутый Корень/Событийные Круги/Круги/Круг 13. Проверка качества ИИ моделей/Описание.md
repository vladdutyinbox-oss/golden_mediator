### Круг 13: A/B-тестирование моделей и личностей (Model A/B Testing)

**Назначение**:  
Организация контролируемых экспериментов по сравнению разных версий моделей и их влияния на поведение AI-личностей. Круг позволяет безопасно тестировать новые fine-tuned или оптимизированные модели на реальных пользователях, собирать метрики качества и принимать обоснованные решения о rollout.

Это один из важнейших инструментов для непрерывного улучшения качества продукта без риска для всех пользователей.

#### Полный путь запроса

1. **Пользователь / Разработчик** заходит в **Model Studio** (`model-management-service`) и создаёт новый A/B-эксперимент:
   - Выбирает базовую модель и одну или несколько экспериментальных версий
   - Выбирает группы пользователей или конкретные личности для тестирования
   - Устанавливает процент трафика (например, 5% → 20% → 100%)
   - Определяет целевые метрики (качество ответа, удовлетворённость, latency и т.д.)

2. Запрос проходит через `api-gateway` → `command-handler`  
   → публикуется событие **`experiment.create.command`**

3. **`model-management-service`** получает команду и создаёт эксперимент:
   - Сохраняет конфигурацию эксперимента
   - Публикует событие **`experiment.created`**

4. **`feature-flag-service`** получает событие и создаёт динамический feature flag:
   - Настраивает правила таргетинга (по user_id, persona_id, процент rollout)
   - Публикует событие **`feature.flag.created`**

5. **Traffic splitting**:
   - При каждом новом запросе в чат (`user.message.sent`) `api-gateway` и `inference-router` запрашивают значение флага
   - `feature-flag-service` возвращает, какую версию модели использовать для данного пользователя/личности

6. **`inference-router`** направляет запрос на нужную версию модели (`rust-inference-service`)

7. **`evaluation-service`** в фоновом режиме:
   - Собирает метрики по всем вариантам эксперимента
   - Проводит сравнительный анализ (качество, скорость, пользовательские сигналы)
   - Публикует событие **`experiment.metrics.collected`**

8. **`model-management-service`** агрегирует результаты и публикует финальное событие  
   → **`experiment.results.published`**

9. Разработчик в Model Studio видит дашборд с результатами эксперимента и принимает решение о победившей версии (rollout или откат).

### Основные события Круга 13

**Входящие команды (триггеры):**
- `experiment.create.command`
- `experiment.update.command`
- `experiment.stop.command`

**Внутренние события:**
- `experiment.created`
- `feature.flag.created`
- `experiment.metrics.collected`
- `experiment.analysis.completed`

**Исходящие события:**
- **`experiment.results.published`** (основное)
- **`model.version.recommended`** (рекомендация по rollout)
- **`experiment.ended`**

### Особенности Круга 13 (SOTA 2026)

- **Progressive Rollout** — постепенное увеличение процента пользователей (1% → 5% → 25% → 100%).
- **Multi-variant Testing** — поддержка одновременного тестирования нескольких версий моделей.
- **Automated Decision Making** — система может автоматически рекомендовать победившую версию на основе метрик.
- **Safety Nets** — мгновенный откат (kill switch) при обнаружении деградации качества.
- **Persona-level Testing** — возможность тестировать разные модели именно на конкретных личностях.
- **Integration with Observability** — все метрики эксперимента видны в реальном времени в Grafana и Model Studio.

Этот круг замыкает петлю улучшения продукта: от генерации новых версий моделей → их тестирования → принятия решения о выпуске.
