### Ключевое взаимодействие:  
**«Динамическое обновление и применение версии модели в процессе генерации ответа»**

Это взаимодействие критически важно, потому что именно оно связывает управление моделями (Model Studio) с реальным инференсом, обеспечивая возможность бесшовного rollout новых версий моделей, A/B-тестирования и персонализации без остановки сервиса.

### Подробное описание взаимодействия (SOTA 2026)

#### Контекст и паттерны системного дизайна

В SOTA 2026 это взаимодействие строится на следующих ключевых паттернах:

- **Event-Driven Configuration Propagation** — изменения конфигурации моделей распространяются через события, а не через прямые вызовы.
- **Dynamic Configuration with Zero-Downtime** — `rust-inference-service` способен подгружать новую версию модели «на лету» без перезапуска.
- **Feature Flag Controlled Rollout** — решение о том, какую версию модели использовать для конкретного запроса, принимается через Feature Flags.
- **Eventual Consistency with Reconciliation** — `model-management-service` публикует событие, а `rust-inference-service` в конечном итоге приводит своё состояние в соответствие.
- **Idempotent Configuration Update** — повторная публикация одного и того же события не приводит к дублирующим действиям.

#### Технологии, используемые в этом взаимодействии

- **Redpanda** (event-bus) + **Protobuf Schema Registry**
- **Temporal.io** — для координации долгоживущих процессов обновления конфигурации
- **Feature Flag Service** (Unleash / Flagsmith)
- **gRPC** + **HTTP** — для синхронных запросов к `model-registry`
- **OpenTelemetry** — полная трассировка конфигурационного потока
- **Rust dynamic loading** (через `libloading` или модульную архитектуру) — для загрузки новой версии модели без рестарта сервиса

#### Подробный поток взаимодействия

1. **Инициация изменения модели**  
   - В Model Studio пользователь или система принимает решение о выпуске новой версии модели (после fine-tuning или оптимизации).  
   - `model-management-service` сохраняет новую версию в `model-registry` и публикует событие **`model.version.published`** (содержит: version_id, model_path, parameters, persona_bindings, feature_flag_key).

2. **Распространение конфигурации**  
   - `model-management-service` публикует событие **`inference.router.config.updated`** в event-bus.  
   - Это событие содержит:
     - `model_version_id`
     - `inference_parameters` (temperature, top_p, context_length и т.д.)
     - `target_personas` или `target_percentage` (для A/B)
     - `feature_flag_rule`

3. **Обработка в inference-router**  
   - `inference-router` потребляет событие и обновляет свою внутреннюю конфигурационную таблицу (в памяти + Redis кэш).
   - При следующем запросе (`context.retrieved`) `inference-router` проверяет Feature Flag и решает, какую версию модели использовать для данного запроса.

4. **Применение в rust-inference-service**  
   - `rust-inference-service` потребляет событие `llm.inference.requested`, которое уже содержит `model_version_id` от `inference-router`.
   - Если версия модели отличается от загруженной в данный момент:
     - Сервис выполняет **dynamic model loading** (загружает новую версию в память, не останавливая сервис).
     - Старые версии могут быть выгружены по LRU или по истечению времени.
   - Генерация ответа происходит уже с новой моделью.

5. **Обратная связь и reconciliation**  
   - `rust-inference-service` периодически (или при загрузке новой версии) отправляет метрики (`inference.metrics.updated`) в `observability-platform`.
   - `model-management-service` может использовать эти метрики для принятия решений о дальнейшем rollout.

### Ключевые SOTA-паттерны в этом взаимодействии

- **Configuration as Event** — конфигурация моделей распространяется как событие, а не через конфиг-файлы или прямые вызовы.
- **Dynamic Hot-Reloading** — `rust-inference-service` поддерживает загрузку новой модели без downtime (state-of-the-art практика 2026 года).
- **Feature Flag + Eventual Consistency** — решение о версии модели принимается на уровне `inference-router`, а применение происходит в `rust-inference-service` асинхронно.
- ** observability-driven Rollout** — решение о полном rollout новой версии принимается на основе реальных метрик из `observability-platform`.

### Роль остальных сервисов в этой интеграции (косвенно)

- **`inference-router`** — критический посредник, который применяет правила Feature Flags и формирует финальный запрос для `rust-inference-service`.
- **`model-registry`** — источник правды о версиях моделей и их артефактах.
- **`observability-platform`** — поставляет данные для принятия решений о rollout.
- **`feature-flag-service`** — определяет, кому и когда давать новую версию модели.

async def publish_new_model_version(self, version_data: dict):
    version = ModelVersion.from_dict(version_data)

    # 2.1 Сохраняем версию в Model Registry
    await self.registry.save(version)                    # → infrastructure/registry/client.py

    # 2.2 Публикуем событие для распространения конфигурации
    await self.kafka.publish(
        topic="inference.router.config.updated",         # Ключевое событие!
        key=version.id,
        value={
            "event_type": "model_version_published",
            "model_version_id": version.id,
            "model_name": version.name,
            "storage_path": version.storage_path,
            "inference_params": version.inference_params,
            "target_personas": version.persona_bindings,
            "timestamp": datetime.utcnow().isoformat(),
            "propagation_required": True
        }
    )

    # 2.3 Логируем метрику
    self._record_metric("model.version.published", {
        "version_id": version.id,
        "propagation_latency_target_ms": 800
    })
```

async def publish(self, topic: str, key: str, value: dict):
    await self.producer.send_and_wait(
        topic=topic,
        key=key.encode(),
        value=json.dumps(value).encode('utf-8')
    )
```

**Шаг 5. Пользовательский запрос (следующий чат)**

Пользователь отправляет новое сообщение:

1. `gradio-frontend` → `api-gateway` → `command-handler`
2. `command-handler` публикует `user.message.sent`
3. `rag-orchestrator` + `memory-service` готовят контекст
4. `inference-router` получает `context.retrieved`
5. `inference-router` проверяет Feature Flag + свою конфигурацию и решает:
   - Какая версия модели должна использоваться для этого пользователя/личности
6. `inference-router` публикует событие **`llm.inference.requested`** с полем `model_version_id: "v4.2"`

---

**Шаг 6. Инференс с новой версией**  
**Сервис:** `rust-inference-service`

- Получает `llm.inference.requested`
- Проверяет, загружена ли уже версия `v4.2`
- Если нет → выполняет **dynamic model loading** (горячая подгрузка)
- Выполняет инференс с новой моделью
- Начинает стримить чанки: `inference.response.chunk`
- По завершении публикует `inference.response.completed`

---

**Шаг 7. Сохранение ответа и памяти**  
**Сервис:** `response-processor`

- Собирает все чанки
- Сохраняет полный ответ в `conversation-service`
- Обновляет память через `memory-service`
- Публикует `memory.updated`
