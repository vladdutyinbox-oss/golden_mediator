### Микросервис: **model-management-service**

#### 1. Общее предназначение
`model-management-service` — это **центральный сервис и backend для Model Studio** — единой панели управления всеми моделями в приложении.

Он выступает как **оркестратор** всего жизненного цикла моделей: предоставляет API для Model Studio, координирует запуск fine-tuning и оптимизации, управляет привязкой моделей к личностям, отслеживает статус экспериментов и обеспечивает удобный интерфейс для разработчиков и продвинутых пользователей.

Особенно важна его роль как **моста** между Model Studio и `rust-inference-service` — он отвечает за то, чтобы в любой момент времени в продакшене использовалась правильная версия модели с нужными параметрами.

#### 2. В каких кругах принимает участие

- **Круг 11** — Fine-tuning моделей (основной)
- **Круг 12** — Оптимизация моделей (основной)
- **Круг 13** — A/B-тестирование моделей и личностей (основной)
- **Круг 3** — Управление личностями (привязка моделей)
- **Круг 1** — Одиночный чат с личностью
- **Круг 2** — Групповой чат
- **Круг 7** — Агентные режимы

#### 3. Какие задачи выполняет

- Предоставляет полный интерфейс и логику **Model Studio**
- Оркестрирует запуск fine-tuning и оптимизации
- Управляет привязкой конкретных версий моделей к AI-личностям
- Контролирует A/B-эксперименты и canary rollout
- Синхронизирует изменения моделей с `inference-router` и `rust-inference-service`
- Отслеживает статус всех задач обучения и оптимизации
- Генерирует рекомендации по улучшению моделей

|--------------------------------------|--------------------|-------------------|-------------------|---------------------|
| **rust-inference-service**           | Критическая        | HTTP / gRPC + Kafka | `inference.router.config.updated`, `model.version.published` | Передаёт информацию о новой версии модели, параметрах и бэкенде. `rust-inference-service` запрашивает актуальную модель при старте и при смене версии |

#### 6. Подробное описание взаимодействия с `rust-inference-service` по этапам и кругам

**В рамках Круга 1 и 2 (Одиночный и групповой чат):**
- Когда пользователь выбирает личность, `model-management-service` через `inference-router` сообщает `rust-inference-service`, какую версию модели и с какими параметрами использовать.
- При публикации события `model.optimized` или `model.fine-tuned` `model-management-service` отправляет `inference.router.config.updated`, после чего `rust-inference-service` подгружает новую версию модели (dynamic loading).

**В рамках Круга 11 (Fine-tuning):**
- После успешного завершения обучения `training-service` уведомляет `model-management-service`.
- `model-management-service` сохраняет новую версию в `model-registry` и отправляет уведомление `rust-inference-service` о доступности новой модели.
- `rust-inference-service` может начать использовать её после подтверждения через Feature Flag.

**В рамках Круга 12 (Оптимизация):**
- После квантизации/прунинга `model-optimization-service` передаёт артефакт в `model-management-service`.
- `model-management-service` обновляет конфигурацию и уведомляет `rust-inference-service` о новой оптимизированной версии (обычно с меньшим потреблением VRAM и выше скоростью).

**В рамках Круга 13 (A/B-тестирование):**
- `model-management-service` совместно с `feature-flag-service` управляет splitting'ом трафика.
- `rust-inference-service` получает от `inference-router` указание, какую версию модели использовать для конкретного пользователя/запроса.

**В рамках Круга 7 (Агентные режимы):**
- Агентные запросы также проходят через `inference-router`, поэтому `model-management-service` обеспечивает, чтобы агент использовал наиболее подходящую (или экспериментальную) версию модели.

### Особенности взаимодействия `model-management-service` и `rust-inference-service`

- **Dynamic Model Loading** — `rust-inference-service` не перезапускается при смене модели. Новая версия подгружается «на лету».
- **Zero-downtime updates** — смена модели происходит без остановки сервиса.
- **Feature Flag controlled** — rollout новой версии модели всегда проходит через feature flags.
- **Version pinning** — для каждой личности можно зафиксировать конкретную версию модели.
- **Metrics feedback loop** — `rust-inference-service` передаёт метрики производительности обратно в `model-management-service` для анализа.

Этот сервис выступает **мостом** между Model Studio (где человек принимает решения) и `rust-inference-service` (где эти решения воплощаются в реальном инференсе).
