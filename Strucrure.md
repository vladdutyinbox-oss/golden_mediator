## 1. Целое приложение
### 1.1 Поведение
#### 1.1.1 Описание и взаимодействие с пользователем

MVP площадки Искуственных Личностей. Можно создавать личности. \
Можно общаться с личностями, которых сам создал. \
Сейчас создание личности только промтом-предустановкой.\
Это шаблоны, которые можно выбирать (например юрист, шутник).\
В будущем можно настраивать личность через свойства психики во \
всех тонкостях её устройства.
Память личности и беседы с ней в VAD графах и векторах.\
Заложена простая эмпатия. Вход и авторизация через отпечаток браузера. \
Общение происходит в классическом в наше время интерфейсе чата. \
Есть поле ввода текста, список личностей, раскатывающееся поле \
бесед с ними. Окно создания личностей. Ответ личности выкатывается \
на лету - через streaming tokens без ожидания полного ответа.

#### 1.1.2 Взаимодействие с разработчиком

Разработчик отслеживает состояние приложения через всевозможные \
дашборды, покрытые всевозможными показателями. Имеется панель \
управления, где можно регулировать задействие мошностей при \
масштабировании. Тут можно поставить автонастройки для kuber \
либо самостоятельно выводить на машины инстансы приложения \
(например при поавной обкатке обновления, когда мы постепенно \
проверяем способность фичи работать в бою под нагрузкой.)

Есть панель управления моделями. Тут их можно подгружать и назначать \
на различные задачи. Пока что задача одна - отвечать, но потом \
будет больше. Можно подгружать размеченные корпуса данных на \
пайплайны дообучения (например дообучение юриста)



### 1.2 Создание и поддержка
#### 1.2.1 Хранение и доставка
Хранится в нескольктх git-репо для возможности бэкапа \
Доставляется так же через git сценарии. Отлаженный CI\CD \

#### 1.2.2 Развертка и расширение
Инстанс может быть развернут docker compose, а грубое \
кратное расширение микросервисов через docker swarm. \
Если требуется развертка отдельных микросервисов с различной \
кратностью и прочие тонкости приспособления к нагрузке - тут всецело \
полагаемся на kubernetes. Обе возможности доступны из командной \
строки

### 1.3 Оценка качества
#### 1.3.1 Показатели уровня

Показатели являются мерой взаимодействия текущего уровня. Рассчитываются \
через соотношения показателей нижних уровней. Предполагаемый \
формат - текущее значение, допустимые границы (с учетом некоторого \
запаса)

Взаимодействие с пользователем:

 - TTFT
 - e2e latency
 - скорость подгрузки веб страниц

Взаимодействие с разработчиком:

 - Стоимость поддержки: за единицу времени
 - Число одновременных пользователей: 

На основании показателей:

Уровень 2....
Уровень 3...

К примеру:


 - использование памяти
 - виды и количество доступных единиц хранилищ
 - Число одновременных сессий
 - доступная мощность вычислителей в Flops
 - количество cpu, gpu с их показателями

#### 1.3.2 Способы измерения и проверки

На дашбордах при полном запуске приложения. Включены тревожные\
упоминания, когда показатели склоняются в ненужную сторону
