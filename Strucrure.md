## Пояснение разметки.

Цель: легко обмениваться пользой с людьми через приложения. \

Для этого нужно уметь быстро проходить от задумки до \
взаимодействия с пользователями. И выполнять это быстро для \
ряда задумок.
Это достигается красотой устройства и простотой 3 действий: \
Разбиваем устройство приложения на уровни от общего к частному. \
От конечного поведения всего приложения к отдельным узлам - хранилищам, \
вычислительным мощностям, видам связи. \
Разбиение на подобные связные самодостаточные уровни. И для каждого \
уровня закладываем возможность 3 действий:

- Создание: Разработка и создание образа. Хранение, доставка и развертка
- Поддержка: поддержание работы приложения в тех или иных \
  границах ключевых показателей при всевозможных условиях. Оценка качества
- Развитие: пуск двух предыдущих действий снова, но из текущего \
  состояния приложения.


Для этого нужны способы управления потоками пользовательских \
взаимодействий и потоком затрат на приложение. Отсюда изобретателю \
польза в деньгах, славе и связях. Пользователям - в услугах, дарующих \
полезные блага. Во взаимном равновесии. \



Растить приложение от отдельных набросков кода до полноценного строения \
теми же 3 видами действий последовательно с возможностью сверять \
на каждом шагу состояние по измеримым показателям.

Как действуем: Разбиваем строение приложения на уровни подходом \
от общего к частному. От целого приложения к его микросервисам и \
их source code.
Каждый уровень делится одинаково на разделы:

- Поведение
- Показатели
- Создание и поддержка

Разделы закладывают в состояние уровня свойства, позволяющие \
над ним выполнять 3 вида действий. Ниже опишем как именно:

1. Поведение. Определяет устройство уровня и одновременно основные \
   составляющие нижнего уровня. Описывает \
   все внутренние и внешние взаимодействия уровня. Внутренние - \
   взаимодействие составляющих между собой. Внешние - взаимодействие \
   текущего уровня с чем либо извне.
   Этот раздел не копает в свойства нижнего уровня, а просто показывает \
   его состав и определет взаимосвязи.
2. Показатели. Это мера внешних взаимодействий уровня. \
   Причем все измеримо и считается прозрачно формулами, где мера \
   внешних взаимодействий есть функция мер внутренних взаимодействий\
   (показателей нижнего уровня.)
3. Действия. Как выполнять 3 вида действий относительно текущего \
   уровня

В итоге получаем, что каждый уровень - самодостаточное промежуточное \
состояние приложения, над которым можно легко выполнять те же 3 вида \
действий.
Каждый новый уровень - следствие предыдущего. Уровни подобны и связаны \
друг с другом. 3 действия над ними легко воспроизводимы в прозрачных \
условиях благодаря показателям.

## 1. Целое приложение

### 1.1 Поведение

#### 1.1.1 Описание и взаимодействие с пользователем

MVP площадки Искуственных Личностей. Можно создавать личности. \
Можно общаться с личностями, которых сам создал. \
Сейчас создание личности только промтом-предустановкой.\
Это шаблоны, которые можно выбирать (например юрист, шутник).\
В будущем можно настраивать личность через свойства психики во \
всех тонкостях её устройства.
Память личности и беседы с ней в VAD графах и векторах.\
Заложена простая эмпатия. Вход и авторизация через отпечаток браузера. \
Общение происходит в классическом в наше время интерфейсе чата. \
Есть поле ввода текста, список личностей, раскатывающееся поле \
бесед с ними. Окно создания личностей. Ответ личности выкатывается \
на лету - через streaming tokens без ожидания полного ответа.

#### 1.1.2 Взаимодействие с разработчиком

Разработчик отслеживает состояние приложения через всевозможные \
дашборды, покрытые всевозможными показателями. Имеется панель \
управления, где можно регулировать задействие мошностей при \
масштабировании. Тут можно поставить автонастройки для kuber \
либо самостоятельно выводить на машины инстансы приложения \
(например при поавной обкатке обновления, когда мы постепенно \
проверяем способность фичи работать в бою под нагрузкой.)

Есть панель управления моделями. Тут их можно подгружать и назначать \
на различные задачи. Пока что задача одна - отвечать, но потом \
будет больше. Можно подгружать размеченные корпуса данных на \
пайплайны дообучения (например дообучение юриста)

### 1.2 Действия

#### 1.2.1 Хранение и доставка

Хранится в нескольктх git-репо для возможности бэкапа \
Доставляется так же через git сценарии. Отлаженный CI\CD \

#### 1.2.2 Развертка и расширение

Инстанс может быть развернут docker compose, а грубое \
кратное расширение микросервисов через docker swarm. \
Если требуется развертка отдельных микросервисов с различной \
кратностью и прочие тонкости приспособления к нагрузке - тут всецело \
полагаемся на kubernetes. Обе возможности доступны из командной \
строки

#### 1.2.3 Способы измерения и проверки

На дашбордах при полном запуске приложения. Включены тревожные\
упоминания, когда показатели склоняются в ненужную сторону

### 1.3 Показатели

- TTFT
- e2e latency
- скорость подгрузки веб страниц

Взаимодействие с разработчиком:

- Стоимость поддержки: за единицу времени
- Число одновременных пользователей:

На основании показателей:

Уровень 2....
Уровень 3...

К примеру:

- использование памяти
- виды и количество доступных единиц хранилищ
- Число одновременных сессий

- количество cpu, gpu с их показателями
